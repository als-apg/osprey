# OSPREY Design System — Designer Handoff

This is the contract between the token source and every browser interface OSPREY
ships (Web Terminal, Artifacts, ARIEL, Channel Finder, Lattice dashboard, the
event-dispatch dashboard, the KNOWLEDGE panel, and the session/safety pages).
Read this before touching a token file or a `.css` file in an enforced interface.

## 1. What lives where

| Path | What it is |
|---|---|
| `tokens/core.json` | Theme-independent primitives: the six color families (`slate`, `teal`, `amber`, `red`, `green`, `blue`, `violet`), `font`, and the `text`/`weight`/`leading`/`space`/`radius`/`z`/`duration` scales. Colors are **never** emitted as CSS variables directly — themes and interfaces reference a color step one hop away via `{color.family.step}` aliases. |
| `tokens/themes/*.json` | One file per theme (6 today). Each aliases `core.json` primitives into semantic names (`bg.primary`, `text.secondary`, `accent.base`, ...). |
| `tokens/interfaces/*.json` | Per-interface extension tokens (`ariel.json`, `artifacts.json`, `channel_finder.json`, `lattice_dashboard.json`, `web_terminal.json` today) — values an interface needs that aren't fleet-wide semantics. |
| `generator/` | The build pipeline: `model.py` (load/parse), `validate.py` (fail-closed checks), `emit_css.py`/`emit_js.py` (renderers), `build.py` (orchestrator + CLI + `--check`). |
| `static/css/tokens.css`, `static/js/tokens.js`, `static/js/theme-boot.js` | **Generated. Never hand-edit.** Regenerated by the one command below; a stray manual edit is overwritten (and CI's `--check` will flag the drift anyway). |
| `static/css/base.css` | Hand-maintained shared reset + primitive element defaults (`html`, `body`, scrollbars, the theme-transition rule). Every browser interface loads this after `tokens.css`. Contains zero color literals and no app-specific selectors. |
| `static/reference.html` | The live token reference page — see §7. |

## 2. The one command

```
python -m osprey.interfaces.design_system.generator.build
python -m osprey.interfaces.design_system.generator.build --check
```

Without `--check` it loads `tokens/`, validates it, and (re)writes the three
generated artifacts. With `--check` it does the same load-and-render but never
writes — it diffs the result against what's on disk and exits non-zero with a
unified diff on any drift. **CI runs `--check`**, so regenerate in the same
commit as any edit under `tokens/`.

## 3. The levers

Every family below is emitted once, at `:root`, and is theme-independent — no
theme can override a `--text-*`/`--weight-*`/etc. value.

| Family | Variable prefix | Steps | Enforcement |
|---|---|---|---|
| Type scale | `--text-*` | `2xs` 8px, `xs` 9px, `sm` 10px, `base` 11px, `md` 12px, `lg` 13px, `xl` 14px, `2xl` 16px, `3xl` 18px, `4xl` 32px | Strict |
| Weight | `--weight-*` | `regular` 400, `medium` 500, `semibold` 600, `bold` 700 | Strict |
| Leading | `--leading-*` | `none` 1, `tight` 1.3, `snug` 1.4, `normal` 1.5, `relaxed` 1.6, `loose` 1.7 | Strict |
| Radius | `--radius-*` | `none` 0, `xs` 2px, `sm` 3px, `md` 4px, `lg` 6px, `xl` 8px, `2xl` 12px, `full` 9999px | Strict |
| Duration | `--duration-*` | `instant` 120ms, `fast` 150ms, `base` 200ms, `slow` 300ms | Strict |
| Spacing | `--space-*` | `0`–`8`, 0px to 64px (4px-based) | Preferred only |
| Z-index | `--z-*` | `base` 0, `dropdown` 100, `sticky` 200, `overlay` 300, `modal` 400, `toast` 500 | Preferred only |
| Fonts | `--font-display`, `--font-mono` | `'Outfit', system-ui, sans-serif` / `'JetBrains Mono', monospace` | Strict |

"Strict" means the hygiene scanner (§5) fails the build on a bare literal in an
enforced interface's `.css`; "preferred only" (spacing, z-index) means the
variable is the recommended spelling for new code but nothing fails if you
write `margin: 16px` — spacing literals are far denser than the other
properties across existing CSS, and z-index frequently encodes legitimate
intra-component micro-stacking a shared scale can't cleanly express.

**Naming caveat:** step names reflect *observed frequency* in the CSS that was
migrated onto this scale, not a designer's judgment about what "base" should
mean. `--text-base` is 11px because 11px was the most common unlabeled
font-size in web-terminal's CSS at migration time — not because 11px is
considered a canonical body size. Renaming a step (or moving where a value
sits on the scale) is a legitimate design decision; make it in `core.json`
and rebuild — never rename by find-and-replace in the generated CSS.

## 4. Themes

Six themes, three families × two modes:

| Family | Dark | Light |
|---|---|---|
| `osprey` (default) | `dark` | `light` |
| `apex` | `apex-dark` | `apex-light` |
| `high-contrast` (WCAG AAA) | `high-contrast-dark` | `high-contrast-light` |

A theme is selected by the `data-theme="<id>"` attribute on `<html>`, applied
pre-paint by `theme-boot.js` to avoid a flash of the wrong theme. The theme
flagged `$extensions.default: true` (currently `dark`) doubles as the `:root`
fallback for any page that never sets `data-theme`.

An interface's extension tokens (`tokens/interfaces/*.json`) can opt a theme
out of authoring its own group via `$extensions.inherits`, e.g.
`{"high-contrast-dark": "dark"}` — the interface then borrows its `dark`
group's values for `high-contrast-dark` instead of duplicating them. This is
for interfaces whose high-contrast twin is purely a color-substitution, not a
structurally different set of extension values.

**To add a theme:** add a new JSON file under `tokens/themes/` with the same
semantic key set as an existing theme of that mode (the validator enforces
key-set parity), give it a `family`/`mode`/`id`/`label` in `$extensions`, then
run the build command. No code changes are needed to wire it into the theme
switcher — it's discovered from the token tree.

Theme switching is entirely client-side (`theme-manager.js`): no server round
trip, no page reload.

## 5. Hygiene rules

Enforced by `tests/interfaces/design_system/test_hygiene.py`, which scans
every `.css`/`.js`/`.html` file under `src/osprey/interfaces/` (plus the
dispatch dashboard):

- **Hardcoded colors — fleet-wide, zero tolerance.** No hex or `rgb()`/`hsl()`
  literal is allowed anywhere in scope. A genuine, permanent exception (a
  print stylesheet, a fixed categorical hue with no token equivalent) gets a
  trailing `/* hygiene-allow-color: <reason> */` comment, or a
  `hygiene-allow-color-start`/`-end` block around a multi-line span.
- **`var()` reference integrity.** Every `var(--name)` must resolve to a
  generator-emitted name or a name declared locally within the same
  interface.
- **No stray token-defining blocks** outside `design_system/static/` — a
  `:root {}` or `[data-theme=...] {}` block may not redeclare a name
  `tokens.css` already emits.
- **Scale literals — scoped, zero tolerance.** A bare `font-size`,
  `font-weight`, `line-height`, `border-radius` (and its longhands), or
  `transition`/`animation` duration literal is forbidden in any `.css` file
  belonging to an interface in `SCALE_ENFORCED_INTERFACES` (currently
  `web_terminal` and `design_system`). A deliberate exception gets a trailing
  `/* hygiene-allow-scale: <reason> */` comment. Two reasons are in active use
  today: `off-scale legacy value, designer to rationalize` (a value with no
  matching scale step, left for a designer to decide whether to fold onto the
  scale or keep as a one-off) and `decorative animation timing, not
  interaction-transition scale` (keyframe animation timing, which is a
  different design concern than the `--duration-*` scale's interactive
  transitions).

**Scanner reach:** the scale check only parses `.css` files and only the
listed longhand/shorthand property names — a `font:` shorthand declaration or
a scale-shaped literal in a `.js`/`.html` file would pass through unseen.
Neither currently exists in `web_terminal` or `design_system`, so this is a
known scope limit, not a known gap.

**On-ramping another interface:** migrate its CSS onto the emitted scale
variables, then add its directory name to `SCALE_ENFORCED_INTERFACES` in
`test_hygiene.py`. The sibling interfaces — `ariel`, `artifacts`,
`channel_finder`, `okf_panel`, `lattice_dashboard` — currently define their
own **local**, interface-prefixed scales (`--ariel-*`, `--art-*`, `--cf-*`,
`--okf-*`, `--ld-*` respectively) rather than consuming the fleet-wide ones —
that's the documented on-ramp path, not a fleet-wide requirement today.

## 7. Preview

`static/reference.html` renders every emitted token — colors, type scale,
fonts, weights, leading, spacing, radii, durations — plus a few composed
specimens (button, badge, card, status pills), all read live off the page's
computed `:root` styles at runtime. There's no hardcoded token list in the
page; add a token to `core.json`/a theme, rebuild, and it shows up
automatically. It includes the `<osprey-theme-switcher>` element, so you can
flip through all six themes in place.

It's served by every interface's shared static-mount helper at
`/design-system/reference.html` — with a Web Terminal (or any other
interface) running locally, open `http://localhost:<port>/design-system/reference.html`.
