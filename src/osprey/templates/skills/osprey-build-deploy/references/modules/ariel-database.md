# Module: ariel

ARIEL is a Postgres-backed semantic-search database that holds enriched logbook entries with vector embeddings. The assistant queries it for "find entries about X" / "what did operators say last week about Y" / "show me previous occurrences of this fault" — questions that keyword search across a raw logbook handles poorly. Enable this when the facility wants the agent to reason over historical operations narrative, not just the most recent few entries.

ARIEL is **a derived view**, not a system of record. Entries are pulled from an upstream source (typically the OLOG module, but other adapters are possible), enriched with embeddings + optional summaries, and indexed for semantic + keyword retrieval. Writes never go directly to ARIEL — they go to the upstream source, and the next sync pulls them in.

**Enabled when**: `modules.ariel.enabled: true` in `facility-config.yml`. Practical prerequisite: an ingestion source must be available (typically `modules.olog.enabled: true`) and an embeddings provider must be reachable (typically `modules.ollama.enabled: true`, or any LLM provider that supports embeddings).

## Architecture

```
   ┌─ upstream source ─┐                                     ┌────────────────────┐
   │  OLOG (or other   │ ── adapter: pull new entries ───▶  │  ariel-sync        │
   │  logbook adapter) │                                     │  one-shot container │
   └───────────────────┘                                     │  on every deploy    │
                                                              └─────────┬──────────┘
                                                                        │
                                                            embedding API call
                                                                        │
                                                                        ▼
   ┌─ embeddings provider ─┐                              ┌─────────────────────────┐
   │  Ollama (local) or    │ ◀── /api/embeddings ─────── │  enrich + write         │
   │  OpenAI/CBORG/etc.    │                              └─────────┬───────────────┘
   └───────────────────────┘                                        │
                                                                    ▼
                                                  ┌──────────────────────────────────┐
                                                  │  Postgres + pgvector              │
                                                  │  (postgresql service if           │
                                                  │   deployment: container,          │
                                                  │   else external DSN)              │
                                                  │                                   │
                                                  │  enhanced_entries (rows)          │
                                                  │  embeddings (vector index)        │
                                                  └────────────┬─────────────────────┘
                                                               │
                                                       ┌───────┴────────┐
                                                       │  ARIEL MCP      │
                                                       │  semantic_search│
                                                       │  keyword_search │
                                                       └─────────────────┘
                                                               │
                                                               ▼
                                                          assistant
```

The MCP server that exposes ARIEL to the assistant is part of the facility's MCP server set (it isn't generated by this module — this module only stands up the database, the sync container, and the schema). What this module owns is the storage, the ingestion pipeline, and the operational lifecycle around them.

## Configuration

Full schema in `references/facility-config-schema.md` § `modules.ariel`. The fields:

```yaml
modules:
  ariel:
    enabled: true
    deployment: "container"                                         # container | external
    dsn: "postgresql://ariel:ariel@ariel-postgres:5432/ariel"        # container default
    # dsn: "postgresql://user:pw@db.example.org:5432/ariel"          # external alternative
    sync_source: "olog"                                              # must reference an enabled module or known adapter
    embeddings_provider: "ollama"                                    # ollama | openai | cborg | ...
```

### `deployment: container` vs `deployment: external`

| Mode | Postgres lives where | DSN points at | When to use |
|------|----------------------|---------------|-------------|
| `container` | The `ariel-postgres` service in this project's compose | `ariel-postgres:5432` (compose DNS name) | Default; simplest; one named volume (`ariel_postgres_data`) to back up |
| `external` | A Postgres the facility already runs (RDS, on-prem cluster, shared dev DB) | The full external DSN, with credentials | Facility has DBA infrastructure, wants centralized backups, or shares the DB across multiple projects |

**Container mode DSN invariant** — the default DSN uses `ariel-postgres` as the host because that IS the compose service key, and Docker/podman DNS resolves service keys to container IPs inside the shared network. If you override the service name in a local compose overlay, override the DSN to match or the sync will fail with `could not translate host name`.

In `external` mode, the `postgresql` service is **not** added to compose. The user is responsible for ensuring the schema + extensions exist on the external DB before first sync (see "Initial sync" below).

### `embeddings_provider`

Whatever value is set here must resolve to a provider that exposes an embeddings API:

- `"ollama"` (most common): reads `${config.modules.ollama.url}` + `${config.modules.ollama.embedding_model}`. Local, fast, free, but the embedding model is fixed by what the Ollama instance has pulled.
- Cloud providers (`"openai"`, `"cborg"`, etc.): use `${config.llm.api_key_env_var}` and the provider's standard embeddings endpoint. Better quality but paid + requires outbound network from wherever `ariel-sync` runs.

Switching providers requires a **re-embed** pass — old vectors won't be comparable to new ones. See "Embeddings provider switching" below.

## What scaffolding adds when this module is enabled

### compose

If `deployment: container`:

```yaml
ariel-postgres:                                       # service key is also the DNS name
  image: pgvector/pgvector:pg16
  container_name: ${config.facility.prefix}-ariel-postgres
  restart: unless-stopped
  ports:
    - "127.0.0.1:5432:5432"     # host-bound to loopback only
  environment:
    POSTGRES_DB: ariel
    POSTGRES_USER: ariel
    POSTGRES_PASSWORD: ariel
    TZ: ${config.facility.timezone}
  volumes:
    - ariel_postgres_data:/var/lib/postgresql/data
  networks: [ ${config.facility.prefix}-net ]
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U ariel -d ariel"]
    interval: 10s
    timeout: 5s
    retries: 5
```

Plus a one-shot `ariel-sync` service that runs `osprey ariel sync` on every deploy:

```yaml
ariel-sync:
  image: ${config.registry.url}/web-terminal:latest   # or the sync image, if separate
  command: osprey ariel sync
  env_file: .env.production
  networks: [ ${config.facility.prefix}-net ]
  depends_on:
    ariel-postgres: { condition: service_healthy }
```

If `deployment: external`, only the `ariel-sync` service is added; `ariel-postgres` is omitted, and `ariel-sync` is given the external DSN via env.

A named volume `ariel_postgres_data` is added to the compose `volumes:` section. **A `--nuke` deploy destroys this volume** and the next sync performs a full re-ingest.

### .gitlab-ci.yml

Nothing specific. The sync container reuses the web-terminal image (which already has OSPREY + the baked project), so no new build job.

### scripts/deploy.sh

After `compose up -d`, the `ariel-sync` one-shot runs to completion and exits. Its output (entries-ingested count, warnings, errors) is captured in the deploy log. `deploy.sh` does not block waiting for sync to finish if the sync is long; the `--clean` and `--nuke` modes both ensure a fresh sync attempt happens.

### scripts/verify.sh

- `ariel_db_reachable`: `psql ${dsn} -c 'SELECT 1'` (uses `psycopg`-style URI parsing).
- `ariel_tables_exist`: query `information_schema.tables` for `enhanced_entries`.
- `ariel_entry_count`: `SELECT COUNT(*) FROM enhanced_entries` — WARNING if 0 (DB up but never synced), OK otherwise.

### .env.template

```
# ARIEL (modules.ariel)
# DSN — for container deployment, the default usually works. For external,
# paste the full URI including credentials.
ARIEL_DSN=${config.modules.ariel.dsn}
```

If `embeddings_provider` is not Ollama, the relevant LLM provider key is already in `.env.template` from the core `llm:` block.

## DSN format

ARIEL uses standard libpq URI syntax:

```
postgresql://<user>:<password>@<host>:<port>/<database>?<options>
```

For container mode, the default `postgresql://ariel:ariel@ariel-postgres:5432/ariel` is plumbed end-to-end (the container's `POSTGRES_USER`/`POSTGRES_PASSWORD` match, and `ariel-postgres` is the compose service key). The credentials are weak by design — Postgres is bound to `127.0.0.1:5432` on the host, so it isn't reachable from off-server. If your facility's threat model requires stronger creds even on loopback, override `POSTGRES_PASSWORD` in `.env.production` and update the DSN in `facility-config.yml` to match.

For external mode, the DSN comes from the DBA. Make sure the user has at least: `CREATE`, `INSERT`, `UPDATE`, `DELETE`, `SELECT` on the schema, plus the right to create the `vector` extension (or have the DBA do it ahead of time).

## Initial sync

On a fresh database (container just started for the first time, or external DB just provisioned):

1. The schema is created automatically by the first `osprey ariel sync` run (it runs migrations against the DSN before ingesting).
2. The first sync pulls **all** entries from `${config.modules.ariel.sync_source}` and computes embeddings for each. This can take from minutes to hours depending on how many historical entries exist and how fast the embeddings provider is.
3. To bound the first sync (useful for testing): `osprey ariel sync --limit N` ingests only the most recent N entries.

Running the sync manually inside the deployed environment:

```bash
ssh ${config.deploy.host} "${config.runtime.engine} run --rm \
  --network ${config.facility.prefix}-net \
  --env-file ${config.deploy.project_path}/.env.production \
  ${config.registry.url}/web-terminal:latest \
  osprey ariel sync"
```

Or from inside an already-running web-terminal container:

```bash
ssh ${config.deploy.host} "${config.runtime.engine} exec ${config.facility.prefix}-web-<user> \
  bash -c 'cd /app/${config.facility.prefix}-assistant && osprey ariel sync'"
```

## Incremental sync

Subsequent syncs are incremental — they fetch only entries created since the last successful run, plus any entries flagged "incomplete" (e.g., missing embeddings because the provider was down on a previous pass). The deploy script runs a sync on every deploy, which is sufficient for many facilities; if the upstream source updates more often than the deploy cadence, schedule a periodic sync via cron, systemd timer, or the `event_dispatcher` module.

The sync command is **idempotent** — running it twice in a row is safe and the second run is fast (only the deltas).

## Schema overview

| Table | Purpose | Notes |
|-------|---------|-------|
| `enhanced_entries` | One row per ingested entry: id, source id, author, timestamp, title, body text, optional AI summary | Primary surface the MCP server queries |
| `embeddings` | Vector column (pgvector) keyed by entry id; one row per chunk if entries are chunked | Indexed with an HNSW or IVFFlat index for ANN search |
| `sync_state` | Bookkeeping: last successful sync timestamp per source, retry counters | Read by the sync command to compute the "since" cursor |

Exact column names and migration history live in OSPREY's ARIEL CLI source. Don't write to these tables manually — go through `osprey ariel ...` commands or the upstream source.

## Verifying the MCP surface works

This module doesn't author the ARIEL MCP server, but the facility's `integration_tests` MCP server should include a check that calls a small ARIEL tool (typically `semantic_search` with a known query) and verifies a non-empty result. From outside:

```bash
curl 'http://${config.deploy.host}:${config.ports.integration_tests}/checks?categories=ariel' \
  | python3 -m json.tool
```

Expected categories at least: `ariel.db_reachable`, `ariel.tables_exist`, `ariel.entry_count`, plus a `mcp_servers.ariel.tools` check that confirms the MCP server exposes its expected tool list.

## Operating the module

### Check ingestion status

```bash
ssh ${config.deploy.host} "${config.runtime.engine} exec ${config.facility.prefix}-ariel-postgres \
  psql -U ariel -d ariel -c 'SELECT COUNT(*) FROM enhanced_entries;'"
```

For per-source breakdown:

```bash
psql ${env.ARIEL_DSN} -c "SELECT source, MAX(created_at) FROM sync_state GROUP BY source;"
```

### Force a full re-sync (re-ingest everything from scratch)

```bash
ssh ${config.deploy.host} "${config.runtime.engine} run --rm \
  --network ${config.facility.prefix}-net \
  --env-file ${config.deploy.project_path}/.env.production \
  ${config.registry.url}/web-terminal:latest \
  osprey ariel sync --full"
```

This is destructive of the sync cursor, not the data — existing rows get updated in place. Useful after fixing a bug in the adapter or after upgrading the embedding model.

### Backup the Postgres volume (container mode)

```bash
ssh ${config.deploy.host} "${config.runtime.engine} exec ${config.facility.prefix}-ariel-postgres \
  pg_dump -U ariel -F c ariel" > ariel-$(date +%Y%m%d).dump
```

For volume-level backup (faster restore, but engine-specific):

```bash
ssh ${config.deploy.host} "${config.runtime.engine} run --rm \
  -v ariel_postgres_data:/data -v /backup:/backup alpine \
  tar czf /backup/ariel-$(date +%Y%m%d).tgz -C /data ."
```

### Restore from a `pg_dump`

```bash
cat ariel-YYYYMMDD.dump | ssh ${config.deploy.host} \
  "${config.runtime.engine} exec -i ${config.facility.prefix}-ariel-postgres \
   pg_restore -U ariel -d ariel --clean --if-exists"
```

### Embeddings provider switching

When you change `embeddings_provider`:

1. Update `facility-config.yml`. Re-scaffold + commit + push + deploy.
2. The new provider is now in use for *new* entries, but old entries' embeddings are in the old vector space and queries that mix old + new vectors will rank poorly.
3. Re-embed everything:
   ```bash
   ssh ${config.deploy.host} "${config.runtime.engine} run --rm \
     --network ${config.facility.prefix}-net \
     --env-file ${config.deploy.project_path}/.env.production \
     ${config.registry.url}/web-terminal:latest \
     osprey ariel reembed --all"
   ```
4. This can be slow — N entries × one embedding call each, throttled by the provider's rate limit.

Plan the switch around a low-traffic window or do it incrementally if the CLI supports `--since`.

### Common failures

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| `postgresql` container won't start | Volume permissions wrong (rootless engine + reused volume from rootful) | Stop, `volume rm ariel_postgres_data`, restart — last resort, since this drops data |
| `ariel-sync` exits non-zero with "connection refused" | Postgres not yet healthy when sync started | Re-run sync; if persistent, check `depends_on: ariel-postgres` healthcheck condition |
| `ariel-sync` exits with "could not translate host name" | DSN hostname doesn't match compose service key | Container mode uses `ariel-postgres` as host; if you overrode the service key in a local overlay, also override the DSN |
| Sync hangs at "computing embeddings" | Embeddings provider unreachable or rate-limited — for Ollama, check `${config.modules.ollama.url}`; for cloud providers, check proxy + key | Verify with `curl` to the provider's `/api/tags` (Ollama) or `/v1/models` (OpenAI-compatible) |
| Entry count stuck at the same number across deploys | Adapter sees no new entries (cursor advanced past everything), or upstream source returned empty | Run `osprey ariel sync --since YYYY-MM-DD` to force a window |
| Semantic search returns garbage | Embeddings provider was switched without re-embed; or `embedding_model` changed under Ollama | Re-embed (see above) |
| Verify check `ariel_entry_count` WARNING (0) | DB is up but ingestion hasn't run, or upstream source has no entries | Run a manual sync; check upstream |

## Disabling

**Heads-up: disabling drops the embeddings.** The Postgres volume is removed by a `--nuke` deploy and orphaned by a regular disable + redeploy (the volume sticks around in the engine's storage but nothing mounts it).

Recommended sequence:

1. **Dump first.** Run `pg_dump` (see "Backup" above) and save the file off the deploy server. Re-ingest is possible from the upstream source, but local enrichment (summaries, manual annotations) is not.
2. Set `modules.ariel.enabled: false` in `facility-config.yml`.
3. Re-scaffold — `postgresql` and `ariel-sync` services drop out of compose; verify checks drop out.
4. `ssh ${config.deploy.host} "cd ${config.deploy.project_path} && ./scripts/deploy.sh --clean"` to stop the running Postgres.
5. To reclaim disk: `${config.runtime.engine} volume rm ariel_postgres_data`. Skip this if you might re-enable.
6. Remove the ARIEL MCP server's tool permissions and `.mcp.json` entry from the facility profile YAML(s) — otherwise the assistant will keep trying to call tools that no longer exist.

Re-enabling is symmetric: re-edit config, re-scaffold, redeploy, and either restore from the dump (`pg_restore`) or run `osprey ariel sync --full` to re-ingest from upstream.
